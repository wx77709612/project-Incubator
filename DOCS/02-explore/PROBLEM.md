## 1. 文档职责

本文档记录 Project Incubator 在 Phase 2 中对核心问题的当前理解。

本文档不定义解决方案、产品设计、自动化机制或实现方式；这些事项只有在 Maker 确认问题定义和研究结论后，才可在后续 Phase 中处理。

## 2. 当前真实工作流

当 Maker 产生一个新想法时，当前真实协作过程通常是：

1. Maker 先与 AI 讨论这个想法如何实现；
2. Maker 与 AI 讨论市面上是否已有对应解决方案；
3. Maker 与 AI 继续讨论方案可行性；
4. Maker 可能直接让 AI 投入代码生成；
5. Maker 通过人肉测试体验生成结果；
6. 在测试和继续协作中发现各种问题；
7. 问题可能来自代码、设计交互、需求理解、协作方式、Git / 文档 / Prompt 管理，也可能来自 Maker 在与 AI 对话过程中受到启发而产生的新想法；
8. Maker 与 AI 讨论如何解决问题，或继续讨论新想法是否值得推进；
9. 在解决一个问题或展开一个新想法的过程中，又发现新的问题或产生更多新想法；
10. 新问题和新想法继续分叉出新的讨论和修改路径；
11. 随着上下文变长，AI 更容易丢失主线、跑偏或重复消耗 token；
12. Maker 很难回到最初那条主脉络，耗时和心智负担持续上升。

## 3. 核心问题

Project Incubator 当前最核心的问题是：

> 在 AI 协助把点子推进为项目结果的过程中，问题发现、问题解决以及受 AI 启发产生的新想法都会不断分叉，逐渐吞掉原始主流程，导致上下文漂移、token 与时间成本上升、项目状态难以恢复，最终让 Maker 很难判断当前究竟是在推进主目标，还是被新问题或新想法牵引着偏离。

这个问题不是单纯的代码生成质量问题，也不是单纯的 Prompt 写得好不好的问题。它更像是一个协作治理问题：当 AI 参与探索、设计、执行、调试和改动时，项目需要一种机制来区分主线、分叉、阻塞、后续任务和已确认决定。

## 4. 痛点链路

当前痛点链路可以概括为：

```text
新想法
→ 与 AI 讨论实现方式和现有方案
→ 判断可行性
→ 让 AI 开始生成代码
→ Maker 人肉测试
→ 发现代码 / 交互 / 需求 / 协作问题
→ 或在与 AI 讨论中受到启发产生新想法
→ 讨论如何解决问题或推进新想法
→ 过程中发现新问题或产生更多新想法
→ 主流程持续分叉
→ 上下文变长，token 和耗时上涨
→ AI 丢失上下文或跑偏
→ Maker 难以还原主脉络
```

## 5. 具体痛点

### 5.1 问题分叉过多

解决一个问题时，常常暴露出新的问题；与 AI 讨论时，也可能受到启发产生新的想法。新问题或新想法可能是重要的，也可能只是局部噪音，但当前缺少明确机制判断：

- 这个问题是否必须现在处理；
- 这个新想法是否应该进入当前主线；
- 这个问题是否阻塞主流程；
- 这个问题是否应记录为后续任务；
- 这个问题是否只是 AI 推断出的“可能有用”事项；
- 这个问题是否改变原始目标、范围或成功标准。

### 5.2 主线容易被覆盖

当对话反复围绕问题和子问题展开后，原始主线会被淹没。Maker 和 AI 可能继续看似忙碌地解决局部问题，但已经不清楚这些动作是否仍服务于最初目标。

### 5.3 上下文与 token 成本持续上涨

随着讨论、文件读取、代码生成、测试反馈和修复记录不断增加，AI 每轮需要处理的上下文越来越重。上下文越拥挤，越容易出现：

- 遗忘前序约束；
- 重复询问或重复解释；
- 使用过期信息；
- 忽略项目已有规则；
- 给出通用建议而非项目特定判断。

### 5.4 项目状态缺少稳定恢复点

如果项目事实只停留在聊天记录中，后续会话很难确定：

- 当前 Phase 是什么；
- 当前真正目标是什么；
- 哪些决定已经确认；
- 哪些只是讨论过但未确认；
- 哪些问题正在处理；
- 哪些问题应延后；
- 当前 Git 状态是否对应一个可审阅的工作范围。

### 5.5 Prompt、文档与 Git 原先缺少统一治理

Maker 过去没有稳定方式管理 Prompt、项目文档和 Git。当前 Project Incubator 已开始用 `TEMPLATES/`、`DOCS/`、`AGENTS.md` 和 Git 规则建立治理，但这些机制仍处在真实使用验证中。

其中，`TEMPLATES/MAKER-TASK-PROMPT.template.md` 已被视为管理 Maker 新任务 Prompt 的候选入口，但 Maker 已确认：它当前还没有成为 Agent 的强制消费入口，这本身就是 Phase 2 需要研究的真实工作流问题。

## 6. 已确认事实

- Project Incubator 是 Maker 自用优先的 Personal Tool；
- 当前真实工作流中，Maker 会先与 AI 讨论想法、实现方式、现有方案和可行性；
- Maker 可能在充分治理前让 AI 直接进入代码生成；
- Maker 会通过人肉测试体验 AI 生成结果；
- 最痛苦、最耗时的环节是“发现问题、解决问题，又发现新问题”，以及“被 AI 启发产生新想法后继续展开”的连续分叉；
- 连续分叉会导致上下文变长、token 和时间成本上升；
- 上下文变长后，AI 更容易丢失主线并跑偏；
- Maker 认可 Phase 2 当前重点应先把问题钉住，而不是提前写解决方案；
- Maker 认可当前不需要把 Phase 2 变成完整产品或竞品研究，但可以参考外部普遍痛点用于补漏。
- Maker 已确认本文档中的核心问题描述准确。

## 7. 当前假设

以下内容尚未被最终确认，只能作为后续研究或设计输入：

- 如果主流程、分叉问题、阻塞问题和后续任务被明确区分，AI 跑偏和上下文漂移可能减少；
- 如果 Prompt 模板能成为 Maker 发起任务的强制消费入口，任务范围可能更稳定；
- 如果每轮协作都有可恢复的状态记录，长会话中的 token 消耗和重复解释可能下降；
- 如果 Git 分支与阶段里程碑绑定，Maker 可能更容易审阅和恢复修改范围；
- 如果问题分叉有明确的“立即处理 / 记录待办 / 请求 Maker 决定 / 停止写入”规则，项目主线可能更不容易被吞掉。

## 8. 待进一步澄清的问题

- “主流程”在 Project Incubator 的不同 Phase 中应如何定义；
- 什么类型的问题必须立刻处理，什么类型的问题应记录为后续任务；
- 当 AI 发现新问题时，它应如何判断是否需要暂停并请求 Maker 决定；
- 当 AI 启发出新想法时，系统应如何判断它属于当前主线、后续任务、待研究假设，还是应暂时拒绝展开；
- Prompt 模板、`PROJECT_STATE.md`、阶段文档和 Git 分支之间应如何分工；
- token、时间和维护成本的可接受阈值是什么；
- Maker 是否需要一个轻量的“问题分叉登记机制”，以及它应属于哪个 Phase 或文档；
- Phase 2 结束前，需要多少外部参考才足够支持下一步设计，而不滑向完整市场研究。
