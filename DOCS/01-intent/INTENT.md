## 1. 文档职责

本文档记录 Project Incubator 为什么做、希望实现什么、不做什么、在何种约束下协作，以及如何判断当前版本达到 Maker 认可的结束状态。

本文档不定义具体产品功能、已有项目接入步骤、版本号细则或自动化实现方案；这些事项留待后续 Phase 按需设计。

## 2. 已确认的主要意图

Project Incubator 的首要意图，是成为 Maker 自用的项目孵化与 AI 协作治理系统。它以从一个点子走向一个项目的过程为主，帮助 Maker 在项目全周期内受控地与 AI 共创、管理项目文档、阶段、上下文和变更，并使项目走向 Maker 认可的结束结果。

这里的“结束结果”不等同于获取收入。它可以是可复用的 Skill、可上线且可能商业化的 App，或一份 Maker 认可、可接受的创作内容。

## 3. 动机与要解决的协作问题

Maker 曾在让创意走向结果的过程中遇到以下管理问题：

- 协作和反复讨论增加 token 消耗；
- 项目推进时不断冒出新点子，容易偏离原目标；
- Git 管理混乱，造成历史修改内容难以追溯或丢失；
- 对话轮次增多后，AI 可能偏离既有上下文或要求；
- AI 可能重新臆想代码、擅自修改内容，甚至进行未获授权的 Git 操作。

Project Incubator 要降低这类混乱，使 Maker 在未来产生新想法或接手已有项目时，都有可持续、可恢复、可审阅的协作流程。

## 4. 目标

### 4.1 主要目标

- 面向新点子，提供从点子到 Maker 认可的结束结果的项目孵化流程；
- 让 AI 协作有清晰的阶段、角色、文档、边界和验收入口；
- 降低项目在上下文、文档、范围与 Git 管理上的混乱；
- 让项目状态和关键决定能够在后续会话中恢复，而不依赖聊天记忆。

### 4.2 次要目标

- 为已有、已推进一段时间的项目提供后续接入能力：基于现有材料协助梳理项目状态、补齐仍未澄清的意图、用户、边界或商业可行性问题；
- 在 Maker 通过真实使用跑通、跑顺流程后，评估是否将其固化为可对外分享的 Skill；
- 为未来可能商业化的具体项目提供受控协作支持，但不把商业化作为默认结果。

## 5. 非目标

- 当前不以 Project Incubator 或未来 Skill 自身商业化为目标；
- 不强迫任何被孵化项目商业化、公开发布，或机械走完全部 Phase；
- 不让 AI 替代 Maker 作出项目方向、范围、阶段切换、验收或版本结束判定；
- 不在当前阶段承诺复杂自动化、完整产品设计或具体实现方案；
- 不把对已有项目的自动推断当作已经确认的项目事实；
- 项目数据与上下文的隐私隔离暂不作为当前范围内的约束，待实际需求出现时再处理。

## 6. 已确认的个人约束

- 每个 Phase 的协作完成后，应检查已有文档是否重复或过度膨胀；重复内容应合并至单一权威来源，以保持文档可读性；
- Skill 应保持轻量的协作成本，避免成为高 token 消耗的专门生码工具；具体 token、时间与维护成本阈值尚未定义；
- AI 可以主动识别可能阶段、生成文档草案并更新状态文档；Maker 审阅并确认最终文档修改；
- 未来被孵化的新项目默认采用 Git，并将 Git 健康检查作为项目检查项；具体 Git 写操作仍遵循项目级授权与验收规则；
- 已有项目接入是次要场景，不能改变以“从点子到项目”为主的项目孵化定位。

## 7. 成功标准与版本判定

### v0.0.1

`v0.0.1` 的最低成功标准是跑通一次真实项目的、与其类型相称的完整孵化流程，并产出该项目的阶段性结果。它不要求流程已经稳定，也不要求恢复上下文、阶段与文档管理等能力在这一版本完成优化。

### v0.x

在 `v0.x` 迭代中，根据真实使用持续改进流程稳定性、上下文恢复、阶段与文档管理等问题。

### v1.0.0

当整个流程已经跑通、没有特别大的问题，并且 Maker 明确认可时，Project Incubator 达到 `v1.0.0`。是否达到 `v1.0.0` 始终由 Maker 明确宣布，不由 AI 自动判定。

## 8. 待后续确认或设计的事项

- token、时间和维护成本的具体阈值；
- 已有项目的接入流程、阶段推断依据、确认机制和文档补全规则；
- 自动识别阶段、生成草案和更新状态文档的具体权限分级；
- 项目数据与上下文的隐私隔离要求；
- 版本号的完整发布与变更规则。
