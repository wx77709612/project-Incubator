## 1. 验证目标

在不接触外部用户、不进行市场或付费验证、不进入产品设计或开发的前提下，观察 Project Incubator 的轻量治理流程在 Maker 的真实任务中是否有助于：

- 保持当前主线；
- 识别并处理新出现的问题或想法分叉；
- 减少重复解释、上下文漂移或任务跑偏。

## 2. 关键假设

如果 Maker 在真实任务开始和推进时使用明确的任务目标、权威文档入口、范围边界与审阅节点，Maker 与 AI 的协作将更容易回到主线，并更容易判断新事项应当立即处理、延后记录还是交由 Maker 决定。

本计划不假定 token、时间或维护成本会达到具体阈值；这些阈值仍未确认。

## 3. 最低成本验证方法

在后续真实自用任务中，Maker 按既有治理流程推进工作。每当出现分叉、重复解释、上下文漂移或需要回到主线的情况时，AI 应先判断新事项属于哪一类，再决定是否写入文档：

- 应立即处理：问题影响当前任务目标、状态入口、权威文档正确性或 Maker 明确要求现在处理；
- 记录后续：问题真实存在，但不影响当前任务继续推进；
- 视为阻塞：问题导致当前任务无法安全继续；
- 请求 Maker 决定：存在方向性取舍、授权边界变化或多种合理处理方式。

随后仅记录以下可观察事实：

- 原任务主目标是否仍可清楚说明；
- 新事项最终被如何处理：立即处理、记录后续、视为阻塞，或请求 Maker 决定；
- 重新读取状态或权威文档后，是否能恢复下一步；
- Maker 是否认为该流程减少了混乱或额外负担。

首轮自用验证结束后，再按证据决定是否创建 `VALIDATION_RESULTS.md`；不预建空白结果文档。

### 3.1 首个方法校准信号

2026-07-22，Maker 针对本计划提出首个反向信号：

- 主目标仍可清楚说明；
- 重新读取状态或权威文档后，可以恢复下一步；
- 流程目前减少了混乱，但是否减少额外负担仍需继续观察；
- 反向信号在于：AI 倾向先把新事项记录为待处理，而不是主动判断它应立即处理、记录后续、视为阻塞，还是请求 Maker 决定；Maker 需要明确要求“现在处理”后，AI 才进入处理。

因此，本计划将“先分类再行动”纳入最低成本验证方法；`PROJECT_STATE.md` 也应保持轻量状态入口，不把阶段门明细或验证进度表扩展成默认必读内容。

## 4. 验证证据与判断

可接受的证据是 Maker 在真实任务中的实际使用记录、任务产物、文档恢复情况、分叉处理结果与明确反馈。外部资料、礼貌性肯定、AI 推断或未经实际使用的功能设想不构成验证证据。

初步支持的信号包括：Maker 能借助现有治理入口恢复主线；分叉事项能获得清楚归类；Maker 明确认可个人使用价值。反向信号包括：流程本身显著增加负担、仍无法恢复主线，或对分叉没有实际帮助。

## 5. 范围与非目标

- 只面向 Maker 自用；不接触外部用户。
- 不做市场规模、竞品、付费意愿或商业验证。
- 不设计产品功能、原型或落地页，不开发实现。
- 验证失败时不强行扩张范围；由 Maker 决定调整、暂停或结束。

## 6. Phase 3 退出判断

在获得首轮真实自用证据后，由 Maker 对照以下任一结果决定下一步：

- 关键假设获得初步支持；
- Maker 确认个人使用价值；
- 验证失败并据此调整项目；
- 验证失败并决定暂停或结束项目。
