## 1. 文档职责

本文档定义 Project Incubator Skill 1.0 中模板应如何形成、验证、实例化和维护。

模板的目标是帮助未来项目快速获得可用的运行入口、状态入口、任务入口和阶段交付物结构，同时避免 AI 为了形式完整预建无价值文档。

## 2. 模板类型

Project Incubator Skill 1.0 至少需要四类模板：

| 模板类型 | 用途 |
| --- | --- |
| Agent 运行协议模板 | 生成新项目根目录 `AGENTS.md` |
| 项目状态入口模板 | 生成新项目 `DOCS/PROJECT_STATE.md` |
| 阶段交付物模板 | 生成各 Phase 按需交付物 |
| Maker 任务 Prompt 模板 | 帮助 Maker 发起新任务或新会话 |

Agent 运行协议模板和项目状态入口模板属于生产启动模板。它们在新项目初始化时使用，直接影响真实项目是否能恢复。

阶段交付物模板属于按需模板。它们只有在对应 Phase 需要交付物时使用。

## 3. 沉淀条件

模板必须来自真实使用，而不是来自文档清单想象。

某类文档满足以下条件后，才可沉淀为模板：

- 已在真实项目阶段中使用；
- Maker 已接受该文档结构或产物；
- 结构中的字段具有跨项目复用价值；
- 字段不会强迫所有项目套用同一深度；
- 能减少重复解释、恢复成本或写入错误；
- 不会把当前项目的临时事实固化到未来项目。

生产启动模板可以在 Phase 4 先完成设计，因为它们是新项目运行的必要入口；模板文件仍应在 Phase 5/6 按任务实现和验证。

## 4. 模板内容规则

模板应保留结构，不预填未确认事实。

模板正文应区分：

- 固定协议：跨项目稳定不变的启动、恢复、写回或验收规则；
- 占位字段：项目名、Maker、项目类型、当前 Phase、目录位置等实例化信息；
- 选择项：不同项目类型、不同 Phase 深度或不同发布方式的可选结构；
- 删除说明：实例化后应删除的模板说明。

模板不应包含：

- 当前 Project Incubator 的阶段事实；
- 某次协作纠偏过程；
- 未经验证的字段；
- 与模板职责无关的长说明；
- 可以由 Skill references 或 Framework 提供的完整规则正文。

## 5. 实例化规则

模板实例化为项目文件时，应遵守：

- 先确认项目类型和当前 Phase；
- 只创建当前需要的文件；
- 未确认字段保留为空或标记为待确认，不写成事实；
- 删除模板说明文字；
- 将新文件状态设为 Draft，除非 Maker 明确接受；
- 更新 `PROJECT_STATE.md` 的权威文档集合；
- 不复制当前孵化器项目的状态事实；
- 不为未来可能需要的 Phase 预建空文档。

## 6. 模板验证

模板提升为稳定模板前，应通过至少一次真实项目验证。

验证时检查：

- 是否减少重复解释；
- 是否帮助 Agent 恢复上下文；
- 是否降低写错权威文档的风险；
- 是否让 Maker 更容易审阅和验收；
- 是否能适配至少一个非当前项目场景；
- 是否没有强制项目进入不需要的 Phase、商业化或开发深度。

模板验证结果应保留在对应 Phase 文档或后续验证文档中，不写入模板本身。

## 7. 与 Skill assets 的关系

未来 Skill 化后，模板可以有两种承载方式：

- 项目本地 `TEMPLATES/`：便于 Maker 查看、修改和按项目演化；
- Skill assets：便于 Skill 安装后直接复制或实例化。

Phase 5 应判断每类模板的承载位置。生产启动模板通常适合放入 Skill assets，并在项目初始化时复制到项目根目录；项目演化后的模板可以保留在项目本地 `TEMPLATES/`。

## 8. Phase 5 输入

Phase 5 可以基于本文档拆解以下任务：

- 设计模板目录结构；
- 创建 `AGENTS.md` 生产模板；
- 创建 `PROJECT_STATE.md` 生产模板；
- 梳理现有阶段文档是否可沉淀为模板；
- 设计模板实例化流程；
- 设计模板验证清单；
- 判断哪些模板进入 Skill assets，哪些保留在项目本地 `TEMPLATES/`。
