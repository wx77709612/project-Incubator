## 1. 文档职责

本文档定义 Project Incubator Skill 1.0 中 Maker 与 AI 如何对话和协作。

本文档不定义界面 UI，也不设计 App 交互控件。当前目标产物是 Skill，因此这里的“交互”指 Maker 与 AI 在不同 Phase、任务、分叉、验收和阶段切换中的协作协议。

## 2. 交互设计目标

Skill 1.0 的交互设计目标是让 Maker 在长期 AI 协作中始终知道：

- 当前正在推进哪条主线；
- AI 此刻扮演什么角色；
- 哪些事项需要 Maker 决定；
- 哪些事项只是 AI 建议或待确认假设；
- 新出现的问题应如何处理；
- 什么时候可以进入下一阶段；
- 什么时候应停下来审阅 Diff 或交付物。

交互不追求复杂仪式感，而是用必要的确认点保护 Maker 的最终决策权和项目主线。

## 3. 启动交互

每次恢复项目时，AI 应先完成状态恢复，再向 Maker 简要说明：

- 当前项目与 Maker；
- 当前 Phase 与 AI 角色；
- 当前主要目标；
- 本轮会读取、创建或更新的文档；
- 当前不处理的事项；
- 验证方式或审阅方式；
- 需要 Maker 决定的内容。

如果本轮只是只读、解释或审查，不创建分支；如果会写入，AI 应在写入前说明 Git 检查结果和实际工作范围。

## 4. Phase 内协作交互

每个 Phase 内，AI 应遵循该 Phase 的 Conversation Contract。通用交互模式是：

1. 恢复状态；
2. 明确本轮主目标；
3. 判断是否需要补读文档；
4. 提出必要问题或方案；
5. 等待 Maker 对关键取舍做决定；
6. 形成阶段交付物草案；
7. 标明 Draft、Active 或后续回收事项；
8. 在阶段门前检查 Exit Criteria。

AI 不应在意图、范围或验收方式不清楚时直接执行。

## 5. 阶段特殊性校准交互

每个 Phase 的 AI 交互都必须先校准当前阶段的特殊性，而不是默认套用固定文档、固定深度或固定流程。

AI 应与 Maker 对齐：

- 当前 Phase 的标准目标是否适合本项目；
- Maker 当前要验收的对象；
- 本 Phase 需要覆盖哪些工作面；
- 哪些文档或产物足以支持下一 Phase；
- 哪些决定必须由 Maker 选择；
- 哪些细节应留到后续 Phase。

如果 AI 与 Maker 对当前 Phase 的目标、深度、交付物或后续衔接有偏差，AI 应把偏差视为关键分叉，请求 Maker 决定，而不是继续写文档或执行。

## 6. 分叉处理交互

当出现新事项时，AI 应先给出分类，再决定行动：

- 立即处理：说明它如何影响当前目标或安全边界；
- 记录后续：说明为什么不影响当前主线；
- 视为阻塞：说明无法继续的原因；
- 请求 Maker 决定：列出关键取舍；
- 拒绝展开：说明它与当前 Phase、非目标或已确认边界的冲突。

AI 应避免使用笼统的“加入待处理”。如果需要记录后续，必须说明记录位置和未来触发条件。

## 7. 高风险动作门禁交互

当 Maker 的指令可能导致高风险结果时，AI 应按结果类型触发门禁，而不是按某个具体词句触发。

AI 的回应必须包含：

- 识别到的风险结果；
- 默认不会直接执行的边界；
- 可供 Maker 选择的安全处理方式；
- 如果需要 AI 执行，必须由 Maker 明确授权的具体动作和范围。

门禁交互应采用以下结构：

```text
我识别到这会影响 [风险结果]。
按当前协作规则，我默认不直接执行该动作。
我可以先提供 [手动步骤 / 方案 / 只读检查结果]。
如果你希望我代为执行，请明确授权 [具体动作、范围、目标]。
```

门禁不应依赖 Maker 的固定表达。只要结果涉及 Git 写操作、Phase 切换、权威文档状态变更、范围扩大、架构边界修改、模板实例化或破坏性操作，就必须先停下来说明。

AI 不应把 Maker 对交付物的接受、对方向的确认、对下一步的口语化推动，解释为可以自动执行高风险动作。此类表达只能触发门禁判断；是否执行仍取决于明确授权和执行前检查。

## 8. 文档审阅交互

当 AI 创建或更新阶段文档后，应：

- 保持文档为 Draft，直到 Maker 接受；
- 报告修改文件和范围；
- 提醒 Maker 在 IDE Diff 中检查；
- 不暂存、提交、推送或合并；
- 等待 Maker 明确确认或提出修改意见。

Maker 接受后，AI 才能将文档状态更新为 Active 或进入阶段门检查。

具体文档写回卫生、纠偏处理和 `PROJECT_STATE.md` 收敛规则由写回规则资产维护。

## 9. Builder 交接交互

当 Phase 5 或 Phase 6 需要交给 Builder 时，Collaborator 必须先给出任务边界：

- 背景和目标；
- 输入、输出和依赖；
- 允许和禁止修改的范围；
- 验收标准；
- 验证步骤；
- 风险和阻塞项；
- 完成后需要更新的文档。

Builder 完成后，AI 必须回到 Collaborator 视角，让 Maker 审阅结果，而不是直接宣布项目完成。

## 10. 阶段切换交互

AI 提出阶段切换前，应输出：

- 当前 Phase Objective；
- 已完成 Deliverables；
- Exit Criteria 逐项检查；
- 未完成项与风险；
- 可供 Maker 选择的下一步；
- 需要更新的状态文档。

只有 Maker 明确决定后，AI 才能更新 `PROJECT_STATE.md` 并切换角色。
