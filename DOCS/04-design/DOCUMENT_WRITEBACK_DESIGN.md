## 1. 文档职责

本文档定义 Project Incubator Skill 1.0 在写入、更新和收敛项目文档时应遵守的设计规则。

本文档解决的问题是：AI 在协作中容易把临时纠偏、单次禁止事项、过程证据、解释性留痕或不属于当前文档的内容写入权威文档，导致文档膨胀、上下文漂移和规则失焦。

## 2. 写回原则

Skill 写回文档时应遵守以下原则：

- 先判断权威归属，再写入；
- 只写对后续恢复、决策、执行、验证或复盘有明确价值的内容；
- 不把聊天过程写成项目事实；
- 不把 AI 推断写成 Maker 确认；
- 不把单次纠偏写成长期规则；
- 不在错误位置留下解释性痕迹；
- 写入后检查文档是否仍符合其权威范围。

## 3. 文档纠偏处理

当 Maker 指出某内容不属于某个文档或某个章节时，AI 应在写回前做归属判断。

允许的处理方式只有三类：

- 删除：该内容没有长期消费价值，或只是临时解释、协作痕迹；
- 移动：该内容有价值，但属于其他权威文档；
- 登记为独立任务：该内容真实重要，但不应混入当前文档，需要后续单独处理。

AI 不应在原位置继续解释“为什么它不属于这里”。文档应保留最终设计结果，而不是保留协作纠偏过程。

## 4. `PROJECT_STATE.md` 写回收敛

`DOCS/PROJECT_STATE.md` 是状态入口，不是过程记录。

写入 `PROJECT_STATE.md` 时，AI 应检查：

- 当前 Phase 是否准确；
- 当前 AI 角色是否准确；
- 当前主要目标是否仍是唯一主线；
- 当前阶段交付物是否只列有效权威路径和状态；
- 当前任务状态是否反映真实工作状态；
- 当前未确认事项是否只保留会影响后续恢复或决策的问题；
- 当前阻塞项是否是真阻塞，而不是普通提醒；
- 下一步是否清楚；
- 下一会话恢复入口是否指向真实文件和当前任务。

阶段交付物基线类状态区块只应保留阶段结果、交付物状态和仍影响恢复的权威路径。它应通过合并、替换或改写现有基线来表达当前状态，而不是追加过程记录。

该类状态区块的写入来源限于：

- Phase 状态变化；
- 交付物路径或状态变化；
- 影响恢复的阶段门结果；
- Maker 明确接受的阶段结论。

写后检查应拒绝以下内容进入状态入口：

- 验证过程细节；
- Maker 某次反馈原文；
- AI 某次错误原因；
- 临时任务过程；
- 单次 Diff 审阅说明；
- 可从 Phase 文档或 Git 历史追溯的细节。

## 5. 边界规则收敛

当 Maker 明令 AI 不要做某事时，AI 不应自动把该事项作为单点禁止项追加到某个长期文档。

AI 应先判断该事项属于哪类已有边界：

- 决策边界：Maker 保留方向、范围、成功标准、验收和阶段切换决定权；
- Phase 边界：当前 Phase 不处理其他 Phase 的目标；
- 范围边界：不因“可能有用”扩大项目类型、公开程度、商业化或实现范围；
- 证据边界：不把推断、建议、礼貌性肯定写成事实；
- 写入边界：不把临时纠偏、过程解释或低复用规则写入权威文档；
- Git 边界：不执行未授权 Git 写操作或破坏性操作。

只有当现有边界无法覆盖，且 Maker 确认该事项具有跨项目复用价值时，才新增长期规则。

## 6. Architecture Decision 候选写回

Architecture Decision 是高权威文档写回，不是普通纠偏记录。

当本轮事项可能影响长期架构边界时，AI 应：

1. 说明它影响哪些跨项目、跨 Phase 或长期治理规则；
2. 判断现有 Architecture Decision 是否已经覆盖；
3. 给出候选写回内容；
4. 等待 Maker 明确确认；
5. Maker 确认后，按 Architecture Decision 结构规则写回。

普通协作偏差、单次错误、临时偏好和低复用规则不得自动上升为 Architecture Decision。

## 7. 写后检查

每次完成文档写入后，AI 应执行写后检查：

- 本次写入是否属于该文档权威范围；
- 是否出现协作纠偏留痕；
- 是否把过程记录写成当前基线；
- 是否新增了单点禁止事项；
- 是否重复维护了其他文档已经负责的事实；
- 是否改变了 Phase、角色、范围或下一步；
- 如果改变了，`PROJECT_STATE.md` 是否已同步；
- 如果未改变，是否避免了无意义状态更新。

写后检查失败时，AI 应先修正文档，再提交给 Maker 审阅。

## 8. Phase 5 输入

Phase 5 可以基于本文档拆解以下任务：

- 设计 `PROJECT_STATE.md` 写后检查清单；
- 设计文档权威范围检查清单；
- 设计 Architecture Decision 候选写回流程；
- 设计文档纠偏处理流程；
- 设计状态入口基线收敛校验；
- 评估是否需要脚本辅助检查文档状态和权威索引。

这些任务应在进入 Build 前形成明确验收标准，否则 Skill 仍可能继续出现纠偏留痕、状态入口流水化和单点规则追加。
