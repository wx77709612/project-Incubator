## 1. 文档职责

本文档定义 Project Incubator Skill 1.0 中项目状态入口应如何设计。

`DOCS/PROJECT_STATE.md` 是每个被孵化项目的当前状态入口。它负责让 Agent 在新会话、恢复会话或任务续作时，快速确认项目当前 Phase、AI 角色、主要目标、权威文档集合、阻塞项、下一步和恢复入口。

本文档不直接替代当前 `DOCS/PROJECT_STATE.md`，也不在 Phase 4 直接生成最终模板文件。最终生产模板应在 Phase 5 拆解并在 Phase 6 实现。

## 2. 定位

`PROJECT_STATE.md` 是高频读取、高频更新、高权威的状态入口。

它不是历史档案，不保存完整对话过程、所有任务记录、旧 Diff 说明或 Git 时间线。

它不是 Agent 运行协议，不规定 Agent 如何执行 Git、如何创建分支或如何结束会话；这些内容由 `AGENTS.md` 负责。

它不是阶段交付物正文，不复制 `IDEA.md`、`INTENT.md`、`PROBLEM.md`、`DESIGN.md` 等 Phase 文档内容，只引用它们的权威路径和状态。

## 3. 与模板的关系

未来被 Project Incubator Skill 孵化出的项目应拥有自己的 `DOCS/PROJECT_STATE.md`。

因此需要 `PROJECT_STATE.md` 生产模板。该模板用于新项目初始化、已有项目接入和状态入口修复。

模板应提供稳定结构，但不得预填未确认事实。实例化时必须由 Maker 或已有项目材料确认以下内容：

- 项目名称；
- Maker 定义；
- 当前 Phase；
- 当前 AI 角色；
- 当前主要目标；
- 当前阶段交付物；
- 当前权威文档集合；
- 当前阻塞项；
- 下一项决定；
- 下一会话恢复入口。

模板文件的创建、命名、放置位置和实例化流程应进入 Phase 5 任务拆解。

## 4. 结构设计

生产版 `PROJECT_STATE.md` 应至少覆盖以下结构：

| 结构 | 职责 |
| --- | --- |
| 文档状态 | 说明本文档的权威范围、状态和更新条件 |
| 基本状态 | 保存当前项目、Maker、Phase、AI 角色、主目标、任务状态和下一项决定 |
| 当前阶段说明 | 保存仍影响当前恢复和阶段推进的阶段背景 |
| 当前 AI 协作契约 | 保存本阶段 AI 应做和不应做的边界 |
| 当前基线 | 收敛已完成阶段、有效治理规则和项目理解基线 |
| 未确认事项 | 保存会影响后续决策、规划或恢复的问题 |
| 阻塞项 | 保存当前无法继续的真实阻塞 |
| 下一步 | 保存恢复后应继续处理的最小入口 |
| 当前权威文档集合 | 保存精确路径、状态和读取要求 |
| 下一会话恢复入口 | 说明下次 Agent 如何恢复，不复制所有上下文 |

## 5. 字段规则

`PROJECT_STATE.md` 的字段应服务恢复，不服务归档。

核心字段规则如下：

| 字段 | 写入规则 |
| --- | --- |
| 当前 Phase | 只能由 Maker 决定或阶段门确认后更新 |
| 当前 AI 角色 | 从当前 Phase 和 Role-System 推导 |
| 当前主要目标 | 只能有一个主线目标 |
| 当前阶段交付物 | 只列有效路径和状态 |
| 当前任务状态 | 反映真实工作状态 |
| 下一项决定 | 指向 Maker 需要确认、接受或选择的事项 |
| 当前权威文档集合 | 必须使用精确路径、状态和读取要求 |
| 未确认事项 | 只保留影响后续恢复、决策或规划的问题 |
| 阻塞项 | 只保留真实阻塞，不保存普通提醒 |

## 6. 基线收敛

`PROJECT_STATE.md` 的基线应按类别收敛，而不是按消息、任务或提交追加。

推荐基线类别包括：

- 阶段与交付物基线；
- 当前有效治理基线；
- 当前项目理解基线；
- 历史追溯边界。

基线只保存仍影响启动、恢复、阶段推进或下一项决定的事实。详细过程保留在对应 Phase 文档、权威文档或 Git 历史中。

## 7. 写入条件

以下变化应更新 `PROJECT_STATE.md`：

- 当前 Phase 变化；
- 当前 AI 角色变化；
- 当前主要目标变化；
- 阶段交付物路径或状态变化；
- 权威文档集合变化；
- Maker 作出影响后续恢复的决定；
- 出现或解除真实阻塞；
- 下一项决定或恢复入口变化；
- Git 闭环状态影响任务状态。

写入时应采用收敛流程：

1. 判断本次内容属于状态变化、权威路径变化、Maker 决定、阻塞变化还是恢复入口变化；
2. 找到对应状态区块；
3. 优先替换或合并现有条目；
4. 只在出现新的长期恢复类别时新增条目；
5. 写入后删除被新状态覆盖的旧表述；
6. 确认详细过程仍由 Phase 文档、权威文档或 Git 历史承载。

如果内容不能归入上述状态变化类型，应保持在对话、对应 Phase 文档或后续任务中，不进入状态入口。

## 8. 写后检查

每次更新后应检查：

- 是否仍只有一个当前主要目标；
- 当前 Phase 与 AI 角色是否匹配；
- 当前阶段交付物是否指向真实文件；
- 权威文档集合是否存在、状态是否正确；
- 未确认事项是否仍影响后续；
- 阻塞项是否是真阻塞；
- 下一步是否可执行；
- 恢复入口是否足够让下一个 Agent 继续；
- 是否出现流水账、纠偏留痕或重复维护事实。

## 9. 与 `AGENTS.md` 的边界

`AGENTS.md` 规定 Agent 如何工作；`PROJECT_STATE.md` 提供 Agent 当前要恢复的项目状态。

`AGENTS.md` 可以要求 Agent 读取 `PROJECT_STATE.md`，但不应保存当前 Phase、当前目标和下一步；`PROJECT_STATE.md` 可以引用 `AGENTS.md` 为启动协议，但不应复制启动协议正文。

## 10. Phase 5 输入

Phase 5 可以基于本文档拆解以下任务：

- 设计并创建 `PROJECT_STATE.md` 生产模板；
- 定义新项目状态入口实例化流程；
- 定义已有项目接入时的状态入口补全流程；
- 设计权威文档集合索引格式；
- 设计状态入口写后检查清单；
- 评估是否需要脚本检查路径、状态和恢复入口。
