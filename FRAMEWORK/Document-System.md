# Document System

> 定义 Project Incubator 的文档分类、用途、生命周期与资产归属

## 1. 文档职责

本文档定义 Framework 文档与具体项目文档的边界，规定文档何时创建、如何维护、由谁消费，以及如何避免聊天记录、重复文档和过期内容造成上下文漂移。

## 2. 核心原则

### 2.1 文档驱动项目

聊天记录不是项目的唯一上下文。影响项目方向、范围、状态、任务或验收的事实必须进入结构化文档。

### 2.2 一个事实只有一个权威来源

同一事实只在一个文档中维护。其他文档可以引用，但不得复制并独立更新。例如：

- 当前 Phase 只在 `PROJECT_STATE.md` 维护；
- 项目范围只在 `SCOPE.md` 维护；
- AI Builder 的仓库级约束只在 `AGENTS.md` 维护；
- 当前代码状态只在 `REPO_CURRENT_STATE.md` 维护。

### 2.3 按需生成

文档只在能够帮助决策、协作、执行、验证或复盘时创建。不是所有项目都需要 PRD、市场分析、技术架构、商业模型或发布策略。

### 2.4 持续更新

文档不是项目启动时的一次性产物。阶段切换、Ticket 完成、重大决策、验证结果或项目状态变化后，相关权威文档必须同步更新。

### 2.5 服务 AI 执行

面向 AI 的执行文档必须明确目标、输入、输出、范围、非目标、依赖、风险、验收标准和验证方法，避免只有愿景或抽象口号。

## 3. 两类资产归属

### 3.1 Framework 文档

Framework 文档属于 Project Incubator，是跨项目复用的系统资产。它们定义规则，不保存某个未来项目的实例状态。

当前 Framework 自举骨架文档为：

| 文件 | 用途 |
| --- | --- |
| `Project-Incubator-Framework.md` | 定义系统定位、核心理念、生命周期与边界 |
| `Phase-System.md` | 定义十个 Phase、阶段门和六项必备结构 |
| `Role-System.md` | 定义 Maker、AI Collaborator、AI Builder 及切换协议 |
| `Document-System.md` | 定义文档分类、用途、生命周期与归属 |
| `Codex-Skill-Specification.md` | 定义未来 Codex Skill 的固化要求 |

未来项目只消费这些规则，不复制整套 Framework。Project Incubator 也不建立第二套 Framework。

### 3.2 具体项目文档

具体项目文档属于对应项目，保存在该项目自己的独立工作区。它们记录项目实例的意图、状态、范围、研究、设计、任务、实现、验证、发布与复盘结果。

具体项目不得把 Framework 文件当作项目文档复制后修改。项目如需说明采用哪一版 Framework，应记录引用信息，而不是维护私有分叉。

本 Project Incubator 仓库不创建 `PROJECTS/` 目录。未来项目工作区的位置与创建机制留待后续人工确认或后续设计，不在本次 Framework 初始化中决定。

### 3.3 项目工作区内部结构

具体项目采用“固定状态入口 + Phase 目录”的混合结构：

- `DOCS/PROJECT_STATE.md` 固定保存在 `DOCS/` 根目录；
- Phase 0–9 的交付物分别保存在 `DOCS/00-idea/` 至 `DOCS/09-archive/`；
- Phase 目录只在当前项目确实需要对应交付物时创建；
- 不预先创建全部空目录；
- 阶段切换不移动旧文档，旧文档保留在其所有者 Phase 目录；
- `PROJECT_STATE.md` 记录当前权威文档集合及准确路径。

Project Incubator 自身也采用这一结构，以验证 Framework 是否能够服务真实项目。

## 4. 具体项目文档分类

### 4.1 状态与治理文档

用途：恢复项目上下文、确定当前 Phase、记录关键约束与后续决定。

典型文档：

- `PROJECT_STATE.md`：项目名称、主要类型、当前 Phase、状态、当前目标、阻塞项和下一项决定；
- `PROJECT_PROFILE.md`：目标使用者、规模、公开程度、商业化程度与推荐路径；
- `INTENT.md`：动机、目标、非目标、约束与成功标准；
- `SCOPE.md`：当前范围与明确排除项；
- `AGENTS.md`：软件项目中 AI Builder 的仓库级技术与执行约束。

`PROJECT_STATE.md` 应保持为紧凑的当前状态入口。它可以记录已形成的当前基线，但不维护按时间追加的完整完成历史。已完成内容应按阶段成果、当前有效治理机制、仍影响恢复和下一步决策的项目事实归类收敛；历史经过、旧版本差异和里程碑细节应从对应权威文档与 Git 历史按需读取。

### 4.2 探索与验证文档

用途：保存问题理解、现有方案、用户证据、关键假设与验证结论。

典型文档：

- `IDEA.md`；
- `PROBLEM.md`；
- `RESEARCH.md`；
- `USER_NOTES.md`；
- `COMPETITORS.md`；
- `MARKET.md`；
- `EXISTING_SOLUTIONS.md`；
- `ASSUMPTIONS.md`；
- `VALIDATION_PLAN.md`；
- `VALIDATION_RESULTS.md`；
- `FEEDBACK.md`。

### 4.3 设计与决策文档

用途：定义第一版成果、体验或内容结构、技术方案、关键取舍与权威范围。

典型文档：

- `README.md`；
- `DESIGN.md`；
- `PRD.md`；
- `UX_FLOW.md`；
- `TECHNICAL_DESIGN.md`；
- `ARCHITECTURE.md`；
- `DATA_MODEL.md`；
- `API_DESIGN.md`；
- `CONTENT_BRIEF.md`；
- `AUDIENCE.md`；
- `CONTENT_STRUCTURE.md`；
- `SCRIPT_OUTLINE.md`；
- `STYLE_GUIDE.md`。

### 4.4 规划与执行文档

用途：将设计转化为范围明确、可独立执行和可独立验证的工作。

典型文档：

- `ROADMAP.md`；
- `MILESTONES.md`；
- `BUILD_PLAN.md`；
- `VERIFICATION_PLAN.md`；
- `TICKETS/` 中的任务文档；
- `REPO_CURRENT_STATE.md`；
- `DEVELOPMENT_LOG.md`；
- 制作记录与版本记录。

### 4.5 发布与迭代文档

用途：支持成果交付、发布、反馈收集和下一轮优先级判断。

典型文档：

- `LAUNCH_PLAN.md`；
- `RELEASE_NOTES.md`；
- `PUBLISH_CHECKLIST.md`；
- `DEMO.md`；
- `MARKETING.md`；
- `BACKLOG.md`；
- `ISSUES.md`；
- `CHANGELOG.md`；
- 更新后的 `ROADMAP.md`。

### 4.6 归档与复盘文档

用途：结束当前项目周期，记录结果、经验、失败原因和可复用资产。

典型文档：

- `RETROSPECTIVE.md`；
- `LESSONS_LEARNED.md`；
- `REUSABLE_ASSETS.md`；
- 最终版 `PROJECT_STATE.md`。

## 5. 文档生命周期

### 5.1 提议

AI 根据当前 Phase、项目类型和待决问题提出需要的文档，并说明用途。按需文档必须说明为什么当前需要。

### 5.2 创建

文档在当前 Phase 确有价值且符合 Maker 指令时创建。创建时必须明确文档职责、权威范围、当前状态和与其他文档的关系。

### 5.3 使用

AI 在会话或执行任务开始前，先读取项目根目录的 `AGENTS.md`，再读取 `DOCS/PROJECT_STATE.md`，然后按照其中的当前权威文档集合加载当前 Phase 所需的最小文档。不得仅凭聊天记忆代替权威文档，也不得通过无目的扫描全部目录猜测当前上下文。

### 5.4 更新

发生以下事件时，应更新相关权威文档：

- Phase 切换；
- Ticket 完成或验收失败；
- 项目目标、范围、主要类型或成功标准经 Maker 确认变更；
- 关键假设得到证据支持或被否定；
- 风险、阻塞项或下一项决定发生变化；
- 发布、反馈、迭代或归档产生新事实。

跨阶段更新必须区分：

- 普通补充：不改变原结论时，更新原权威文档；
- 新阶段结论：在当前 Phase 创建新文档，并引用上游权威来源；
- 推翻前序决定：由 Maker 批准阶段回退或受控变更，再更新原权威文档和所有受影响索引。

更新 `PROJECT_STATE.md` 时，还必须区分状态入口、历史记录与按需读取：

- 状态入口只保留当前恢复和推进所需的有效事实、文档路径、阻塞项、阶段门状态和下一项决定；
- 历史记录保留在原权威文档、对应 Phase 文档和 Git 历史中；
- 按需读取由 `PROJECT_STATE.md` 的权威文档集合提供精确路径，不通过新增默认必读历史文档解决流水账问题；
- 只有当 Maker 明确确认存在长期复盘、审计或跨任务检索需求时，才考虑新增历史文档或索引文档；新增后也必须标记为按需读取，而非普通启动必读。

### 5.5 审核

阶段退出前，AI 应检查：文档是否反映真实状态、是否存在冲突来源、是否有未标记假设、是否包含过期决定，以及是否足以支持下一 Phase。

### 5.6 冻结或归档

不再维护但仍需保留证据价值的文档应标记状态或归档，不应继续被当作当前权威来源。项目归档时，最终状态、复盘和可复用资产必须可独立理解。

## 6. 创建与更新规则

每份项目文档至少应能回答：

- 它解决什么问题；
- 它属于哪个项目；
- 它的所有者 Phase 是什么；
- 它的文档状态是 Draft、Active、Superseded 还是 Archived；
- 它唯一负责什么权威范围；
- 哪些后续 Phase 会消费它；
- 哪些输入支撑其内容；
- 哪些决定已经确认，哪些仍是假设；
- 什么事件会触发更新；
- 哪个后续角色或任务会消费它。

项目文档应使用 `TEMPLATES/DOCUMENT-METADATA.template.md` 定义的通用元数据字段。文档类型模板可以增加字段，但不得删除项目、所有者 Phase、状态、权威范围、消费 Phase、更新条件、依赖文档和最后更新时间。

执行型文档还必须明确：

- 目标与非目标；
- 允许与禁止的范围；
- 输入、输出和依赖；
- 验收标准；
- 验证方法；
- 风险和未解决问题。

## 7. AI 读取顺序

恢复具体项目时，AI 应按以下顺序读取最小必要上下文：

1. 项目根目录的 `AGENTS.md`，确认启动协议；
2. `DOCS/PROJECT_STATE.md`，确认 Maker、当前状态、Phase、AI 角色、目标与下一决定；
3. `SPECS/ARCHITECTURE_DECISIONS.md`，确认不可静默改变的架构决策；
4. `FRAMEWORK/Phase-System.md` 中当前 Phase 的六项规则；
5. `FRAMEWORK/Role-System.md` 中当前角色规则；
6. `PROJECT_STATE.md` 所列当前权威文档集合；
7. 与本轮任务直接相关的设计、范围、Ticket 或验证文档。

`PROJECT_PROFILE.md`、`INTENT.md` 或其他尚未产生的后续阶段文档，不得被当作所有阶段启动的固定前提。AI 应以 `PROJECT_STATE.md` 的实际索引为准。

若索引缺失、路径不存在、权威文档之间冲突或存在多个 Active 来源，AI 必须先报告问题，不得自行选取对执行更方便的版本。

## 8. 文档质量标准

合格文档应具备以下特征：

- 内容清晰、具体、可操作；
- 事实、假设、建议与决定有明确区分；
- 对当前 Phase 和后续消费者有直接价值；
- 不与其他权威文档重复维护同一事实；
- 对范围、验收和验证有足够约束；
- 更新状态与真实项目状态一致；
- 不包含无依据的项目扩张。

## 9. Framework 变更治理

Framework 的变更属于 Project Incubator 自身的架构或方法变更，不能由未来具体项目直接覆盖。任何会改变六项架构决策、Phase 定义、角色边界、文档归属或主要流程原则的修改，都应先回到最高设计依据并获得 Maker 明确确认。

具体项目产生的经验可以形成 Framework 改进建议，但建议与已批准规则必须分开记录；未经确认，不得直接成为新的 Framework 规则。

## 10. 文档所有权与跨阶段复用

### 10.1 所有者 Phase

每份阶段文档都有且只有一个所有者 Phase。所有者 Phase 负责形成、解释和治理该文档的权威范围，但不限制其他 Phase 读取。

### 10.2 跨阶段引用

后续 Phase 通过准确路径引用上游文档，不复制正文。例如 Planning 和 Build 可以继续读取 `DOCS/04-design/SCOPE.md`，无需在自己的目录创建范围副本。

### 10.3 权威文档索引

`DOCS/PROJECT_STATE.md` 必须维护当前权威文档集合，至少标明内容类别、准确路径、文档状态和本轮是否必读。AI 应按索引读取，不依赖文件名猜测或递归搜索。

### 10.4 替代与失效

当新文档正式替代旧文档时：

- 旧文档状态改为 Superseded；
- 旧文档指向新权威路径；
- `PROJECT_STATE.md` 更新索引；
- 下游任务和文档引用同步检查；
- 不删除仍具有决策或证据价值的历史文档。

## 11. 模板治理

- `TEMPLATES/` 保存可复用结构，不保存具体项目事实；
- 通用元数据模板适用于所有权威项目文档；
- 不创建一个覆盖所有文档类型的万能正文模板；
- 各文档类型模板应根据对应 Phase 的真实使用逐步建立；
- 模板首次使用时可以是 Draft；
- 只有经过真实项目使用、Maker 审阅并确认有价值后，才能标记为稳定模板；
- 未找到模板不阻止当前阶段工作，AI 应根据 Phase 规则创建最小草案，并避免补造事实；
- 模板变化不自动改写已经存在的项目文档。
